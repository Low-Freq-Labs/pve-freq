### pfSense Post-Firmware-Update Runbook


**3. Verify/install sudo:**
```
pkg info sudo
/usr/local/bin/sudo -l -U svc-admin
```
If missing: `pkg install -y sudo`, then verify `/usr/local/etc/sudoers` contains `#includedir /usr/local/etc/sudoers.d` and `/usr/local/etc/sudoers.d/svc-admin` contains `svc-admin ALL=(ALL) NOPASSWD: ALL`. Both chmod 440.

```
ifconfig lagg0 | grep mtu
ifconfig lagg0.5 | grep mtu
ifconfig lagg0.10 | grep mtu
ifconfig lagg0.25 | grep mtu
ifconfig lagg0.66 | grep mtu
ifconfig lagg0.2550 | grep mtu
```

If any show 1500, fix at runtime:

```
ifconfig igc2 mtu 9000 && ifconfig igc3 mtu 9000 && ifconfig lagg0 mtu 9000
```

## Lessons Learned

These are hard-won rules from real incidents. Do not ignore them.

### 1. WireGuard Endpoint = Sacred (Session 4)
Before ANY VIP/WAN IP change: check pfSense VPN → WireGuard → Server for the bound endpoint. If the IP being changed matches, do NOT proceed. Always have LAN/console access before touching firewall or VPN remotely.

### 2. vmbr0 Address Is Sacred (Session 11)
`vmbr0` on pve01 MUST keep `10.25.0.26/24`. Management VLAN belongs on `vmbr0v2550`, NEVER on vmbr0 itself or `vmbr0.2550` (split-brain bug, Lessons #13/#14).

### 3. NFS fstab Must Have `_netdev,nofail` (Session 12)
Without `_netdev`: mount before networking → hang. Without `nofail`: mount failure stops boot → console-only recovery.

### 4. Asymmetric Routing Breaks NFS (Session 13)
VMs on VLAN 5 routing to TrueNAS via switch SVI causes asymmetric paths. Fix: Static routes on TrueNAS for return traffic via switch SVI.

### 5. VLAN 10 Gateway Must Be Switch SVI (Session 11)
VLAN 10 pfSense rule blocks NFS traffic. Switch SVI routes NFS directly via L3.

### 6. SQLite Cannot Run on NFS (Session 15)
NFSv3 file locking is unreliable. All configs/DBs must be on local disk.

### 7. Fleet DNS Standard: 1.1.1.1 / 8.8.8.8 (Session 15, updated S065)
All systems use 1.1.1.1 (primary) + 8.8.8.8 (secondary). VLAN 5 VMs lock resolv.conf with `chattr +i`. VM 103 DNS managed by Gluetun. pfSense uses local Unbound (127.0.0.1) + upstream 1.1.1.1/8.8.8.8. No systems point at local Unbound (10.25.0.1) as resolver.

### 8. Jumbo Frames Must Match End-to-End (Session 1)
Switch, hosts, VMs, TrueNAS all must have matching MTU. Mismatched MTU = silent packet drops.

### 9. Guest Agent API for VM Root Access (Session 12)
Use Python + Proxmox REST API for `agent/exec`. `qm guest exec` breaks with flags in PVE 9.

### 10. Proxmox VM Create: Use pvesh on Target Node (Sessions 6-7)
Cross-node API calls have format issues. SSH to target node and use `pvesh create` directly.

### 11. pfSense Config Requires Root (Session 6)
config.xml is root:wheel. Non-root PHP writes silently fail.

### 12. iDRAC LOM + VLAN Tagging (Session 10)
R530 iDRAC shares eno1 (LOM1). Trunk port with native VLAN 1 + tagged 2550. `racadm set iDRAC.NIC.VLanID 2550`.

### 13. VLAN Sub-Interface Split-Brain (Session 17)
`vmbr0.2550` competes with Proxmox's `nic0.2550`. Kernel delivers tagged frames to `nic0.2550` first. Fix: use `vmbr0v2550`.

### 14. VLAN 2550 Split-Brain Breaks Corosync (Session 17)
Same bug as #13 broke corosync ring0 on management VLAN addresses.

### 15. ZFS Bind Mount Ordering on Boot (Session 17)
Bind mount fires before ZFS datasets mount. Fix: `x-systemd.requires=zfs-mount.service`.

### 16. Always Eject ISO After OS Install (Session 17)
Leftover ISOs cause VM start failures when the ISO file is deleted.

### 17. Parallel NFS Writes Can Overwhelm Soft Mounts (Session 34)
Stagger NFS-heavy operations. Soft mounts drop rather than hang (correct behavior).

### 18. AMD Polaris GPU Needs vendor-reset (Session 34)
RX580 doesn't properly reset via VFIO. Needs `vendor-reset` kernel module on Proxmox host.

### 19. Never Change pfSense Interface via GUI on LACP Members (Session 35)
"Apply Changes" restarts ALL interfaces → LACP PDU disruption → switch err-disable. Use runtime `ifconfig` and persist to config.xml manually.

### 20. Add err-disable Recovery to Cisco Switches (Session 35)
No auto-recovery by default. Configure: `errdisable recovery cause link-flap` + interval.

### 21. Never Run `rc.reload_interfaces` with LACP Active (Session 36)
Bounces ALL interfaces. Use runtime `ifconfig` or full reboot.

### 22. Host Route Trick for DR Recovery (Session 36)
When /24 route is stuck, add `/32 host route` via DR port to override.

### 23. SSH ProxyJump for Switch Access (Session 36)
`ssh -o ProxyCommand="ssh -W %h:%p pfSense" switch`

### 24. Cisco IOS Config via SSH stdin Pipe (Session 36)
`echo "conf t\n...\nend\nwrite memory" | ssh -T switch`

### 25. FreeBSD sed Doesn't Support `\t` (Session 36)
Use perl for config.xml edits needing tabs.

### 26. pfSense tcsh Breaks Bash Quoting (Session 36)
Use base64-encoded scripts: `echo '<base64>' | b64decode -r | sudo sh`.

### 27. Clean Up Stale IPs on LACP Members (Session 36)
Check member interfaces for stale IPs after LACP recovery.

### 28. Runtime WAN IP Migration Doesn't Update pf (Session 36)
pf rules stay bound to original interface. Must edit config.xml and reboot.

### 29. Set Up DR WAN Before Changing Primary WAN (Session 36)
Configure WANDR VPN failover BEFORE touching primary WAN.

### 30. Duplicate IPs Cause ARP Flapping (Session 36)
Remove IPs from old interface before/immediately after migration.

### 31. Remove `lacp rate fast` for Stability (Session 36)
Fast LACP rate causes repeated err-disable during reboots. Use normal rate (30s).

### 32. Errdisable Recovery Interval 30s (Session 36)
Combined with watchdog cron, max auto-recovery ~90s.

### 33. Routed /29 Block Goes to Primary WAN (Session 36)
Upstream routes 69.65.20.56/29 to lagg1. VIPs on WANDR won't receive inbound.

### 34. Multi-Homed Servers Need Policy Routing (Session 39)
Single static route causes pfSense to drop replies from "wrong" interface. Use `ip rule add from <source-IP>` per-interface routing.

### 35. pf Can't Forward Sustained NFS Writes Between VLAN Sub-Interfaces (Session 40)
Route inter-VLAN NFS via switch L3 instead of pfSense.

### 36. `mountpoint -q` Unreliable After Lazy Unmount (Session 40)

### 37. Docker fwmark + Policy Routing Collision (Session 42)
Docker 29+ doesn't create CONNMARK/fwmark rules. Need explicit Docker bridge routes in policy table + prio 201 rule for container reply packets.

### 38. pfSense if-bound States Don't Match Cross-Interface Traffic (Session 42)
Add explicit pass rules on destination VLAN for return traffic to VPN subnet.

### 39. pvestatd Hangs on Unreachable NFS Storage (Session 43)

### 40. Docker Hardlinks Require Unified Mount (Session 46)
Use single volume mount containing both source and destination directories.

### 41. WSL SMB Mounts Break on TrueNAS Interface Changes (Session 46)
Workaround: SSH to TrueNAS directly. CIFS auto-reconnects once routing restored.

### 42. NFS bindip Changes Flush Policy Routing Tables (Session 46)
Verify `ip route show table 100` after any NFS bindip change. Re-add if empty.

### 43. midclt Static Route Operations Trigger Full Network Applies (Session 47)
Can temporarily drop OTHER routes. Verify routing after `midclt call staticroute.delete/update`.

### 44. pfSense Self-Originated Traffic Must Source from VIP (Session 48)
ISP only routes VIP /29 traffic. Transit /28 IPs are dropped. All pfSense services needing internet must use `outgoing-interface: 69.65.20.58`.

### 45. config.xml custom_options Must Be Base64-Encoded (Session 48)
PHP calls `base64_decode()`. Raw text is silently ignored.

### 46. /etc/sshd Generates sshd_config at Boot (Session 48)
No config.xml toggle for PermitRootLogin. Direct edit required. Firmware updates overwrite.

### 47. NTP Requires rc.d Startup Script for VIP Binding (Session 48)
pfSense regenerates ntpd.conf at boot. rc.d script at `/usr/local/etc/rc.d/ntpd_vip_fix.sh`.

### 48. filter.inc Crash: Rules Missing Interface Tags (Session 49)
All firewall rules in config.xml must have `<interface>` tags.

### 49. Always Backup Config Externally (Session 49)
All local pfSense backups were lost after config restore. SCP to TrueNAS/WSL before risky changes.

### 50. LAGG Config Uses Comma-Separated Members (Session 49)
`<members>ix2,ix3</members>` — flat string, not nested XML.

### 51. PHP Shell for Config Changes (Session 49)
Menu option 12. `config_set_path()` + `write_config()`. Use `/etc/rc.filter_configure` for filter reload.

### 52. SSH Authenticates Against config.xml Users (Session 49)
`pw useradd` alone won't work. Use `local_user_set()` via PHP.

### 53. check_reload_status Auto-Restarts (Session 49)
Can't be killed permanently. Fix the rules instead.

### 54. Unbound custom_options Lands Outside server: Block (Session 50)
Prefix with `server:\n` before base64 encoding.

### 55. Option 3 Only Updates master.passwd (Session 50)
Does NOT update config.xml. Password reverts on reboot. Sync via PHP.

### 56. Repeated Failed SSH Triggers virusprot Table (Session 50)

### 57. FreeBSD su Requires wheel Group (Session 50)

### 58. write_config() Triggers Filter Reload (Session 50)
SSH may drop for ~30s. Don't panic.

### 59. FreeBSD sed Doesn't Support \t in Insertions (Session 50)
Use awk: `awk -v line="$LINE" '...' file > tmpfile && mv tmpfile file`.

### 60. pfSense Has TWO SSH Blocking Tables (Session 51)
Check both `virusprot` AND `sshguard`. Flush with `-T delete <ip>`.

### 61. local_user_set_password() May Silently Fail (Session 51)
Workaround: Generate hash with `password_hash()`, write directly to config array.

### 62. Docker host Network Mode Bypasses Port Binding (Session 51)
Only iptables on host can restrict. `netfilter-persistent save`.

### 63. sshpass Bash History Expansion with ! (Session 51)
Use `sshpass -f /tmp/.svc-pass` to avoid shell interpretation.

### 64. deluser Fails After usermod UID Change (Session 52)
Use `gpasswd -d USER GROUP` instead.

### 65. pfSense pkg+https:// Uses mTLS Client Certificates (Session 54)
Package repos use mutual TLS with short-lived client certs (~12h). If `pkg update` returns 400 errors, run `/usr/local/sbin/pfSense-repo-setup` to refresh certs from Netgate. Do NOT add FreeBSD repos or `IGNORE_OSVERSION` overrides — that is wrong.

### 66. pfSense Firmware Updates Wipe /opt/ But Preserve /usr/local/ (Session 54)
Scripts in `/usr/local/sbin/` and `/usr/local/etc/rc.d/` survive upgrades. Scripts in `/opt/` do NOT. Config.xml entries (cron, shellcmd) survive. Always place custom scripts in `/usr/local/sbin/`. See Post-Firmware-Update Runbook.

### 67. pfSense VLAN MTU Does Not Auto-Inherit from Parent at Runtime (Session 54)
Setting lagg0 to MTU 9000 does NOT propagate to VLAN sub-interfaces at runtime. Must set each VLAN interface explicitly with `ifconfig lagg0.X mtu 9000` AND persist `<mtu>9000</mtu>` in each VLAN definition in config.xml.

### 68. pfSense ntpd `interface listen` Must Cover All Client VLANs (Session 55)
`interface ignore wildcard` blocks ALL interfaces, then `interface listen <IP>` selectively enables. If a client is on a VLAN whose gateway IP is not in the listen list, NTP queries time out even though the kernel routes them locally. VM 103 (dirty VLAN 66) couldn't reach 10.25.0.1 or 10.25.255.1 due to pf rules, but could reach 10.25.25.1 (storage VLAN, direct L2). Fix: add all VLAN gateway IPs that NTP clients need (`10.25.0.1`, `10.25.25.1`, `10.25.255.1`) to `ntpd_vip_fix.sh`.

### 69. WSL fstab SMB Mount Needs `noauto` When Behind VPN (Session 55)
WSL `mount -a` at boot with CIFS mounts to VPN-only targets (10.25.25.25) hangs for ~10 minutes when WireGuard isn't up yet. `nofail` prevents hard failure but doesn't prevent the hang. Fix: add `noauto` to fstab, mount manually or via session bootstrap after VPN is confirmed up.

### 70. vendor-reset Module Requires Kernel Header Version Match (Session 56)
The vendor-reset DKMS module compiles against a specific kernel version's headers. When a PVE node reboots into a newer kernel (e.g., 6.17.9→6.17.13), the module is missing for the new kernel. If headers for the new kernel aren't yet in the Proxmox repo, the module can't be rebuilt. Workaround: GPU passthrough works after a clean host boot (BACO reset mechanism), but vendor-reset is needed for VM stop/start cycles without host reboot. The `asm/unaligned.h` header was renamed to `linux/unaligned.h` in kernel 6.12+ — patch required for vendor-reset compilation.

### 71. Proxmox pve-apt-hook Protects Metapackage Dependencies (Session 56)
`apt purge` on Proxmox nodes is blocked by `/usr/share/proxmox-ve/pve-apt-hook` if removing a package would also remove the `proxmox-ve` metapackage. Packages like `lzop` (pve-manager dep), `postfix` (pve-manager dep for notifications), and `isc-dhcp-client` (pve-container dep for LXC DHCP) cannot be purged. Remove packages individually to identify which are protected vs. safe to remove.

### 72. VM 104 NTP Needs Storage VLAN Server Like VM 103 (Session 57)
VM 104 on pve03 has its default route via VLAN 5 (ens18, 10.25.5.34). VLAN 5 blocks RFC1918 traffic, so NTP to pfSense LAN (10.25.0.1) times out. Fix: set `NTP=10.25.25.1` (pfSense storage VLAN interface) in `/etc/systemd/timesyncd.conf`. Same root cause and fix as VM 103 (S055). **Rule: Any VM with default route through VLAN 5 or VLAN 66 must use storage VLAN (10.25.25.1) for NTP.**

### 73. TrueNAS midclt SMB Update Requires `purpose` Field (Session 57)
`midclt call sharing.smb.update` rejects payloads with just `options` — returns "You must set `purpose` if you set `options`". Always include `"purpose": "DEFAULT_SHARE"` (or the share's current purpose) alongside `options` changes. The `hostsallow`/`hostsdeny` fields live under `options`, not at the top level of the share object.

### 74. VM 103 Policy Rule 201 Caused Asymmetric Routing for DNAT'd WebUI Traffic (Session 57)
S042 added `ip rule add from 172.16.0.0/12 to 10.25.100.0/24 lookup 200 prio 201` to route Docker container replies to WireGuard through dirty VLAN. This broke WebUI access (port 8080 on management VLAN) from WireGuard clients: the SYN arrived on ens19 (management) but the SYN-ACK left via ens18 (dirty) because the container's reply (src 172.18.0.2) matched rule 201 before conntrack reverse-DNAT. pfSense dropped the SYN-ACK (state mismatch — wrong interface). **Fix:** Removed rule 201 from runtime and `/etc/network/interfaces`. Container replies to WireGuard now use the main routing table (`10.25.100.0/24 via 10.25.255.1 dev ens19`). Torrent traffic unaffected — peers are public IPs, not 10.25.100.0/24. **Rule: DNAT'd traffic from different VLANs cannot share a single policy routing rule — routing decision happens before conntrack reversal.**

### 75. AMD VAAPI Requires format=nv12,hwupload Filter Chain (Session 58)
Unlike NVIDIA (which handles pixel format conversion internally via hwaccel_output_format), AMD VAAPI requires explicit software-to-hardware surface upload: `-vf format=nv12,hwupload`. Without this, h264_vaapi fails with "Impossible to convert between the formats supported by the filter 'Parsed_hwupload_vaapi'". The full working command for AMD RX580: `tdarr-ffmpeg -hwaccel vaapi -hwaccel_device /dev/dri/renderD128 -i input -vf "format=nv12,hwupload" -c:v h264_vaapi -qp 22 output`. Do NOT use `-hwaccel_output_format vaapi` — that's the NVIDIA pattern.

### 76. Tdarr Encoder Test vs Actual Plugin Execution (Session 58)
Tdarr runs a built-in encoder compatibility test on node startup. The test for h264_vaapi fails (shows `h264_vaapi-true-false` in logs) because the test pipeline doesn't include the required `-vf format=nv12,hwupload` filter. Despite this failed test result, GPU workers DO successfully pick up and execute custom plugins that use h264_vaapi — the encoder test result does not block plugin execution on GPU workers. Without GPU workers enabled, VAAPI transcode jobs sit in queue with "requires GPU worker". **Rule:** Keep GPU transcode workers ≥1 for VAAPI jobs. The encoder test failure is cosmetic — it doesn't prevent actual transcoding.

### 77. Prowlarr fullSync Overwrites Radarr/Sonarr Indexer Settings (Session 58)
Prowlarr periodically syncs indexer configurations to connected apps (Radarr, Sonarr). This sync OVERWRITES per-app indexer settings like minimum seeders. If you set min seeders to 15 in Radarr/Sonarr but leave Prowlarr at 1, the next fullSync resets them. **Rule:** Always set indexer-level settings (min seeders, priorities) at the Prowlarr level first. Per-app overrides are temporary.

### 78. Two Interfaces on Same Subnet Causes Default Route Hijack (Session 59)
When two interfaces share the same /28 subnet (e.g., lagg1 WAN at 100.101.14.2/28 and igc0 WANDR at 100.101.14.3/28), FreeBSD's ARP resolution for the shared gateway (100.101.14.1) consistently picks the **last-configured** interface at boot. This hijacks the default route to the wrong interface. The pfSense `defaultgw4` setting and `route add -ifp` do NOT override this — FreeBSD resolves the gateway MAC via ARP on whichever interface has the "freshest" connected route. **Rule:** Avoid two interfaces on the same L3 subnet when one must be the default route. If unavoidable, use the /32-flip trick (Lesson #79) or Gateway Groups.

### 79. /32-Flip Trick Pins Default Route to Correct Interface (Session 59)
When FreeBSD hijacks the default route to the wrong interface due to shared subnets (Lesson #78), temporarily set the offending interface to /32 (which removes its /28 connected route), then add the default route (which now resolves via the only remaining /28 interface), then restore the /28 (the route stays pinned). This works because FreeBSD doesn't re-resolve the gateway's ARP entry when the second /28 is re-added — it keeps the existing route table entry. **Warning:** This is a runtime trick, not a config.xml solution. Must be applied after every reboot via boot script.

### 80. /32 on igc0 Without Gateway Won't Crash filter.inc (Session 59)
S049 learned that /32 on igc0 WITH a gateway crashes `filter.inc` on reboot (Lesson #48). However, /32 on igc0 WITHOUT a gateway item in config.xml survives reboot cleanly — no filter errors, no config breakage. The crash was caused by the gateway entry referencing an interface with a /32 subnet, not the /32 itself. **Rule:** /32 on WANDR is safe as long as no gateway is assigned to it in config.xml. But /32 prevents upstream from delivering traffic (ARP can't resolve), so it's not useful for DR.

### 81. pfSense WebGUI Blocks Overlapping Subnets on Different Interfaces (Session 59)
pfSense WebGUI returns "IPv4 address X is being used by or overlaps with: [interface]" when trying to assign the same /28 subnet to two different interfaces (e.g., WAN and WANDR both on 100.101.14.0/28). This is a GUI-only validation — config.xml accepts the overlap. **Fix:** Edit config.xml directly via SSH (perl one-liners) for multi-WAN on the same transit subnet. Similarly, VIPs require a parent interface with a configured IPv4 address — can't add VIPs to interfaces with "None" for IPv4.

### 82. pfSense reply-to Handles Transit Return Traffic Correctly (Session 59)
Even when the default route points to lagg1, pfSense's `reply-to` mechanism in pf ensures that traffic arriving on igc0 (WANDR) gets its replies sent back out igc0 with the correct source IP. This means DR WireGuard works regardless of default route — packets arrive on igc0, WireGuard processes them, and replies go out igc0 via reply-to. The default route only matters for self-originated traffic and NAT'd outbound from internal VLANs.

### 83. pfSense Gateway Group Items Use Pipe-Delimited Format (Session 59)
Gateway group members in config.xml use pipe-delimited strings: `<item>GATEWAYNAME|TIER|VIRTUAL_IP</item>`, NOT nested XML sub-elements. pfSense's `write_config()` serializes PHP arrays as nested XML, which breaks `gwlb.inc` (`explode('|', ...)` fails on arrays). When creating gateway groups via PHP API, use string values like `"WANGW|1|interface_address"` for each item. If nested XML was written, fix with perl to convert back to pipe-delimited format. Gateway Group as `defaultgw4` adds pf `route-to` rules for all WAN-facing pass rules, routing transit traffic explicitly through the correct interface regardless of the kernel routing table.

### 84. /32 on WANDR (igc0) With Non-Default Gateway Survives Reboot (Session 60)
S049 showed /32 + DEFAULT gateway crashed `filter.inc`. S060 proved /32 + NON-DEFAULT gateway (WANDRGW inside Gateway Group) boots cleanly — no filter errors, pf generates all rules correctly. The key difference: S049 had WANDRGW as the default gateway, S060 has WANDRGW as Tier 2 in a Gateway Group with `defaultgw4=WAN_FAILOVER`. **Rule:** /32 on a DR interface is safe as long as its gateway is non-default (e.g., inside a Gateway Group).

### 85. /32 Interface Cannot ARP for Non-Connected Hosts — pf reply-to Breaks (Session 60)
When igc0 has /32 (100.101.14.3/32), it has no connected network — only a host address. pf's `reply-to (igc0 100.101.14.1)` directive forces reply packets out igc0, but the kernel can't ARP for 100.101.14.1 on igc0 because the address is outside igc0's /32 "subnet". Packets are silently dropped. WireGuard handshake initiations arrive (pf state created, tcpdump shows inbound), but replies never leave. **Fix:** Add a host route (`route add -host 100.101.14.1 -iface igc0`) which tells the kernel 100.101.14.1 is directly reachable at L2 on igc0. This enables ARP without affecting the default route (already resolved to lagg1). **Rule:** /32 on a multi-homed WAN interface requires a host route for the gateway to support pf reply-to.

### 86. Host Route for Gateway Does Not Affect Already-Resolved Default Route (Session 60)
Adding `route add -host 100.101.14.1 -iface igc0` when the default route (`default 100.101.14.1 UGS lagg1`) is already in the routing table does NOT change the default route's interface. FreeBSD's default route was resolved at creation time with lagg1's connected /28 route, and the host route is a separate, more specific entry. Internet traffic continues via lagg1 while pf reply-to can ARP for the gateway on igc0 via the host route. Confirmed across 3 reboots.

### 87. Gateway Monitor IP Must Account for Host Routes (Session 60)
WANGW's default monitor target is its gateway IP (100.101.14.1). After adding a host route for 100.101.14.1 via igc0, the monitor's ICMP pings resolved to igc0 instead of lagg1 (verified via `route get` and pf states). This caused asymmetric routing: ping exits igc0, reply returns on lagg1, pf drops the reply (if-bound state mismatch). WANGW showed 100% loss / down. **Fix:** Changed WANGW `<monitor>` to `8.8.8.8` in config.xml. This routes monitor pings via the default route (lagg1), proving actual internet connectivity — which is more useful than gateway L2 reachability anyway. **Rule:** When adding host routes that override gateway reachability paths, check if any gateway monitor targets are affected.

### 88. pfSense pf `flags S/SA` Only Applies to TCP — ICMP Passes Through (Session 60)
pfSense firewall rules with `flags S/SA keep state` are TCP-specific. The `flags S/SA` modifier only filters TCP packets (checking SYN flag). Non-TCP protocols (ICMP, UDP) bypass the flags check entirely and match on the rule's other criteria (source, destination, interface). A rule like `pass in from 10.25.100.0/24 to <MGMT_NETWORK> flags S/SA` passes TCP SYN packets AND all ICMP AND all UDP from VPN to management. No separate ICMP rule is needed for ping to work through pfSense WireGuard rules. **Rule:** Don't create separate ICMP allow rules on pfSense — the existing per-VLAN TCP rules already pass ICMP implicitly.

### 90. Proxmox NIC Naming Via systemd .link Files — Not udev Rules (Session 61)

### 91. LSIO Container NFS Bind Mount: Pre-Create Directories (Session 61)

### 96. Multi-NIC VMs and Asymmetric Routing from VPN (Session 66)

Public VLAN IPs (.5.x) are for internet-facing services, not admin access.

### 98. Tdarr File Size Check: QP 25 on BluRay Sources Compresses to ~30-35% (Session 66)

### 111. Sonarr/Radarr Language CF Doesn't Block Dual-Language or Multi-Sub Releases (Session 68)

### 113. Huntarr Settings API Replaces Entire Section (Session 68)

### 114. Hardlink Counts Determine Real Disk Usage in Complete Dirs (Session 68)
`du -sh` on download complete directories reports inflated totals because it counts files at their full size regardless of hardlink status. Files with `links=2` share an inode with the library copy — deleting from complete/ frees zero actual disk space (link count goes 2→1, data persists). Only files with `links=1` (orphaned after Tdarr re-encode broke the hardlink) consume extra space. In S068: `du` reported 610GB in complete dirs, but only 208GB was truly reclaimable (links=1 files). **Rule:** Before cleaning download dirs, check `stat -c '%h' <file>` — links=2 means hardlinked (zero extra cost), links=1 means orphaned (safe to delete, frees real space). Use `find <dir> -type f -links 1` to find only orphaned files.

### 115. SABnzbd Speed Depends on Provider Retention vs Content Age (Session 68)
SABnzbd download speed is not a network/config bottleneck — it's driven by article availability on the usenet provider. Frugal Usenet (primary, priority 0) showed 97.7% article failure rate (2.1M articles tried, only 48K succeeded) because content aged 87-1160 days exceeds Frugal's retention window. Blocknews (backup, priority 25) had better 36% failure rate but only activates as fallback. `bandwidth_perc` setting caps at % of detected line speed (80% was leaving 20% unused). `queue_limit` controls simultaneous download items — too many (20) spreads connections thin across failing articles. **Fix:** Set `bandwidth_perc=100`, lower `queue_limit` to 5 for immediate improvement. Long-term fix: add a high-retention backbone provider (Newshosting/Eweka with 5800+ day retention) as primary. **Rule:** When SABnzbd is slow, check article completion % per server before blaming network — the bottleneck is usually content availability, not bandwidth.

### 116. SABnzbd host_whitelist Breaks After Container Recreation (Session 68)
SABnzbd's `host_whitelist` in `sabnzbd.ini` contains the Docker container hostname (e.g., `297a363ac594`). When a container is recreated (`docker compose up -d` after config change), a new hostname is assigned (e.g., `406c2d9753b9`). Restart SABnzbd after edit. **Rule:** Always include `localhost` in SABnzbd host_whitelist to survive container hostname changes.

### 117. VAAPI AV1 Decode Not Supported on AMD Polaris / RX 580 (Session 69)
AMD Polaris GPUs (RX 580, GCN 4th gen) have NO AV1 hardware decoder. AV1 decode support starts with RDNA 2 (RX 6000 series). Using `-hwaccel vaapi` on AV1 content causes ffmpeg to fail at the decode stage. The VAAPI transcode plugin must detect AV1 codec (`av1` in `codec_name`) and either skip (`processFile: false` to fall through to CPU plugin) or use software decode + GPU encode. Similarly, 10-bit content (common in anime, HDR) fails with `format=nv12` (8-bit only). Detection: `pix_fmt` containing `10` (e.g., `p010le`, `yuv420p10le`) or `bits_per_raw_sample >= 10`. **Fix for 10-bit:** Use `-vaapi_device /dev/dri/renderD128` (software decode → GPU encode) instead of `-hwaccel vaapi -hwaccel_device /dev/dri/renderD128` (full hardware pipeline). **Rule:** When writing VAAPI transcode plugins, always check codec and bit depth — not all content can use the full hardware decode pipeline.

### 118. Sonarr Season 0 Specials Waste Huntarr Search Cycles (Session 69)

### 119. Tdarr cruddb API Uses POST, Not PUT (Session 69)
Tdarr's `/api/v2/cruddb` endpoint returns 404 for PUT requests (`Route PUT:/api/v2/cruddb not found`). The correct method is POST for both reads and writes to internal JSON databases (NodeJSONDB, StagedJSONDB, StatisticsJSONDB, etc.). POST body format: `{"data": {"collection": "<db_name>", "mode": "update", "docID": "<key>", "doc": {...}}}`. Auth via `x-api-key` header using the `seededApiKey` from `Tdarr_Server_Config.json`. **Rule:** Always use POST for Tdarr cruddb operations — the API is POST-only despite being used for read and write operations.

### 120. SABnzbd `pre_check` Prevents Phantom Queue Orphans in Sonarr/Radarr (Session 69)
When SABnzbd downloads a NZB with too many missing articles, `fast_fail` aborts the download and places it in SABnzbd history with status **"Queued"** (not "Failed"). Sonarr/Radarr only recognize "Failed" as a trigger for `autoRedownloadFailed` — "Queued" is treated as still-active. This creates phantom queue entries in the arr services that show `status=downloading, trackedDownloadStatus=ok` even though SABnzbd already aborted. Manual cleanup required. **Fix:** Enable `pre_check: 1` in SABnzbd config (`set_config&section=misc&keyword=pre_check&value=1`). Pre-check verifies article availability via HEAD requests BEFORE downloading. If articles are missing, the job is properly marked as "Failed" in history, which triggers the arr services' auto-redownload pipeline. Combined with `fail_hopeless_jobs: 1` (already enabled) and `autoRedownloadFailed: True` (both arr services), the full self-healing chain is: SABnzbd pre-checks → marks Failed → arr detects → auto-removes → auto-searches alternative release. **Rule:** Always enable `pre_check` on SABnzbd when using arr service integration — without it, aborted downloads create orphan queue entries that require manual intervention.

### 121. QEMU Guest Agent Shows net0 IP as Primary in PVE UI (Session 70)
Proxmox displays the IP address from `net0` (ens18 inside the VM) as the primary address in the VM summary and guest agent info. If the "wrong" IP shows up in PVE, the fix is to swap which VLAN tag is on `net0` vs `net1` in the VM config (`qm set`), then correspondingly swap the IPs assigned to ens18/ens19 inside `/etc/network/interfaces`, and reboot. The MAC addresses and the IPs themselves don't change — only which physical NIC carries which VLAN. Docker containers bound to specific IPs survive the swap because the IPs are unchanged. **Rule:** To control which IP PVE displays as primary, put the desired VLAN on net0.

### 122. TrueNAS @includedir Wiped by Middleware — Third Occurrence (Session 70)

### 123. HEVC→H264 Transcoding GROWS Files — Skip HEVC ≤1080p (Session 72)
HEVC (H.265) is 30-40% more efficient than H.264 at the same quality. Converting HEVC to H.264 via VAAPI (QP 25) or libx264 (CRF 23) consistently produces LARGER output files. This caused 14 of 25 Tdarr error states — files transcoded, grew in size, hit the size check upperBound, and errored. The `deleteCacheAnyStageError: 1` setting then deleted the cache, recording `newSize: 0`. **Fix:** Skip HEVC files at or below target resolution (1080p). Only transcode HEVC when downscaling from 4K+ (saves space via resolution reduction, not codec change). Same logic applies to AV1 (even more efficient than HEVC). **Rule:** Never transcode a modern efficient codec (HEVC/AV1) to an older one (H.264) at the same resolution — it wastes compute and grows files. Only transcode for resolution reduction or from inefficient codecs (MPEG-2, H.264 high-bitrate) to save space.

### 125. PVE Cluster Requires Root SSH Between Nodes on Corosync IPs (Session 74)
PVE migration, console proxying, and storage replication all use SSH between nodes as root. The migration command always does an initial SSH to the **corosync address** (`.0.x`) even when a migration network is set — this first hop establishes the tunnel, then data flows over the migration network. `PermitRootLogin no` in SSH config blocks all inter-node cluster operations. `PermitRootLogin prohibit-password` is the correct hardening — allows key-based root login (used by PVE cluster) while blocking password-based root login. SSH must listen on BOTH the management VLAN (`.255.x`) and the corosync/cluster VLAN (`.0.x`). Use `/etc/ssh/sshd_config.d/dc01-hardening.conf` drop-in files for consistency (main sshd_config may be overwritten by updates). **Rule:** When hardening PVE SSH, always use `prohibit-password` for root, never `no`. Always listen on both mgmt and cluster IPs.

### 124. TrueNAS Middleware DB is the Permanent Fix for Sudoers (Session 73)
TrueNAS middleware owns `/etc/sudoers` and regenerates it from its internal database. Any manual additions (`@includedir`, hand-edited lines) get wiped on regeneration — observed 3 times (S063/S066/S070). The permanent fix is to store sudo commands IN the middleware database itself via `midclt call user.update <uid> '{"sudo_commands": ["cmd1", "cmd2", ...], "sudo_commands_nopasswd": ["cmd1", "cmd2", ...]}'`. Middleware then writes these commands into `/etc/sudoers` as its own managed config — it cannot wipe entries it generated itself. After migration: remove `@includedir /etc/sudoers.d`, delete `/etc/sudoers.d/*.conf` files, and remove any cron/POSTINIT band-aids. Verify with `sudo -l` for each account. **Rule:** Never use `/etc/sudoers.d/` files on TrueNAS SCALE — they require `@includedir` which middleware will wipe. Always use `midclt call user.update` to manage sudo permissions natively.

### 126. pfSense Sudoers Get Wiped Periodically — Always Verify After Updates (Session 75)
pfSense wipes `/usr/local/etc/sudoers.d/` contents during firmware updates or system resets. Unlike TrueNAS (where we migrated to middleware DB), there is no native pfSense mechanism to persist custom sudoers. The only fix is to redeploy after each wipe. If missing, redeploy via base64 method.

### 127. iDRAC 7/8 Only Supports RSA SSH Keys — Not Ed25519 (Session 75)

Use a single RSA key for accounts that need both Linux + iDRAC access (avoid managing two key types).

### Lesson 75. Debian 13 Yescrypt Breaks SSH Password Auth (Session 63)
Debian 13 (Trixie) defaults to yescrypt (`$y$`) for password hashes. SSH PAM authentication silently fails with yescrypt hashes — `sshpass` returns exit code 5 (wrong password) even though the password is correct. Fix: `echo "user:password" | chpasswd -c SHA512` to convert to `$6$` hashes. Check `grep '^\$y\$' /etc/shadow` on all Debian 13 installations.

### 93. Cisco Config Register Is Stored in NVRAM Immediately (Session 65)
`config-register 0x2102` writes to NVRAM as soon as the command is executed — not via `write memory`. Both `reload` and power cycle read the new value from NVRAM. `show version` displays "Configuration register is 0x2142 (will be 0x2102 at next reload)" meaning the current boot session used 0x2142 but NVRAM already contains 0x2102. The switch is safe from unexpected reboots once the command has been run. Always follow with `write memory` to save the running-config to startup-config as well.
